# 代码规范

> 本项目（Godot 4.6 / GDScript）的统一编码规范。**所有新代码必须遵守，已有代码逐步统一。**
> 基线：Godot 官方 GDScript 风格指南 + 本项目约定。

已确定的三条总原则：
1. **类型标注**：严格全标注 —— 所有变量、参数、返回值都带类型（能推断的用 `:=`）。
2. **注释语言**：中文。
3. **旧代码**：顺手统一成本规范（含 `event_bus.gd` / `sound_manager.gd`）。

---

## 1. 文件与编码

- 编码 **UTF-8**，换行 **LF**（见 `.editorconfig` / `.gitattributes`，提交时 Git 会归一化）。
- 缩进用 **Tab**（不是空格）。
- 文件末尾留一个空行。
- 字符串统一用**双引号** `"..."`。

---

## 2. 命名规范

| 对象 | 规则 | 示例 |
|---|---|---|
| 脚本文件 / 文件夹 | `snake_case` | `base_device.gd`、`scripts/devices/` |
| 场景文件 `.tscn` | `snake_case` | `main.tscn`、`generator.tscn` |
| `class_name` | `PascalCase` | `BaseDevice`、`BaseStateMachine` |
| 节点名（场景里） | `PascalCase` | `Player`、`DayManager`、`SmokeBar` |
| 函数 / 方法 | `snake_case` | `add_repair_cost()` |
| 变量 | `snake_case` | `smoke_progress` |
| 常量 `const` / `enum` 成员 | `CONSTANT_CASE` | `DAY_LENGTH`、`NORMAL` |
| 枚举类型名 | `PascalCase` | `DeviceState` |
| 信号 | `snake_case`，动词过去式或名词 | `device_state_changed` |
| 私有成员 / 方法 | 前缀 `_` | `_setup()`、`_pool` |
| Autoload 单例 | `PascalCase` | `EventBus`、`Game` |
| 节点组 group | `snake_case` 字符串 | `"player"`、`"devices"` |
| 布尔变量 | 用 `is_/has_/can_` 前缀 | `is_open`、`has_fuel` |

**EventBus 事件名**：`snake_case`，统一动词/名词，集中记录（见第 7 节）。例：`"device_state_changed"`、`"show_settlement"`。

## 3. 格式规范

- **冒号后留一个空格，前面不留**：`var x: int`、`func f(a: int) -> void:`。
- **逗号后留空格**：`func f(a: int, b: int)`。
- **运算符两侧留空格**：`a + b`、`x = 1`、`i += 1`。
- **括号内侧不留空格**：`foo(a, b)`，不是 `foo( a, b )`。
- 一行只做一件事，不写 `;` 串多句。
- 行宽建议 ≤ 100 字符，超了就换行。
- **空行**：函数之间空一行；类内逻辑块之间可空一行；`func` 与上一段之间空一行。
- `->` 返回类型前后各一空格：`func f() -> void:`。

```gdscript
# ✅ 正确
func add_repair_cost(value: int) -> void:
	today_repair_cost += value

# ❌ 错误（冒号无空格、逗号无空格）—— event_bus 旧写法，需统一
func push_event(destination:String,payload=[]) -> void:
```

## 4. 类型标注（严格全标注）

**规则**：所有变量、参数、返回值都必须有类型。能让引擎推断的用 `:=`，推断不出的显式写 `: 类型`。

```gdscript
var speed: float = 220.0          # 显式
var dir := Vector2.ZERO           # := 推断（等价于显式 Vector2）
var best: BaseDevice = null       # null 推断不出，必须显式

func f(a: int, b: String = "") -> bool:
	return true

# 循环变量也标类型
for dev: Node in get_tree().get_nodes_in_group("devices"):
	pass

# 信号声明带参数类型
signal device_state_changed(device: BaseDevice, state: int)
```

要点：
- **无返回值也写 `-> void`**。
- `@export` 变量必须带类型（否则编辑器面板报错）：`@export var speed: float = 220.0`。
- `@onready` 也标类型：`@onready var fsm := get_parent() as BaseStateMachine`。
- 字典/数组尽量用类型化容器：`var states: Dictionary[StringName, BaseState] = {}`、`var items: Array[int] = []`。

## 5. 注释与文档（中文）

- **文件/类文档**：脚本顶部用 `##` 写该脚本职责（一两句）。
- **行内注释**：用 `#`，解释“为什么”而非“做什么”。
- 公开方法若行为不直观，上方加 `##` 说明。
- 不写废话注释（`i += 1  # i 加一` 这种禁止）。

```gdscript
extends Area2D
class_name BaseDevice
## 所有设备的统一接口。子类只重写行为，不改框架。

## 猴子篡改设备
func tamper() -> void:
	...
```

## 6. 脚本内部结构顺序

严格按此顺序排列（官方推荐）：

```
01  class_name（如有）
02  extends
03  ## 文件文档注释
04  signal
05  enum
06  const
07  @export 变量
08  普通 var
09  @onready var
10  _ready() / _process() / _physics_process() 等生命周期函数
11  公开方法
12  _私有方法（_前缀）
```

## 7. 架构约定（必须遵守）

### 7.1 全局单例 Autoload
| 名称 | 文件 | 职责 |
|---|---|---|
| `EventBus` | `global/event_bus.gd` | 事件总线，模块解耦通信 |
| `SoundManager` | `global/sound_manager.gd` | 音效池播放 |
| `Game` | `global/game_state.gd` | 全局状态 + 配置常量 + 键位注册 |

### 7.2 模块通信：优先 EventBus，不要跨模块直接调用
- 跨系统、一对多、解耦场景 → 用 `EventBus.push_event()` / `subscribe()`。
- 父子节点、紧耦合的内部调用 → 可直接引用，但**子节点不要反向硬引用父节点的具体类型**（用信号往上抛）。
- **所有事件名集中登记在本节末尾的事件表**，新增事件先加表再用，避免拼写漂移。

```gdscript
# 发布
EventBus.push_event("device_state_changed", [self, new_state])
# 订阅（通常在 _ready）
EventBus.subscribe("device_state_changed", _on_device_changed)
```

### 7.3 状态机
- 用 `BaseStateMachine` + 子节点状态（继承 `BaseState`）。
- 状态切换调用 `transition_to(&"StateName")`，状态名用 `&"..."` StringName。
- 每个状态一个脚本，放对应目录（如 `scripts/day/`）。

### 7.4 设备统一接口 `BaseDevice`
- 所有设备继承 `BaseDevice`，**只重写**：`_setup()`、`_device_process()`、`_on_tamper()`、`_on_repair()`、`_draw()`、`inspect()`。
- 不在子类里重新发明状态/费用逻辑，统一走基类。

### 7.5 全局状态集中在 `Game`
- 天数、金钱、进度、今日费用等跨场景数据放 `Game`。
- **配置数值（Demo 阶段）写死成 `const`**，集中在 `Game` 或各自脚本顶部，不散落在代码中段（魔法数字）。

### 📋 EventBus 事件表（新增请在此登记）
| 事件名 | payload | 说明 |
|---|---|---|
| `device_state_changed` | `[device: BaseDevice, state: int]` | 设备状态变化 |
| `show_settlement` | `{base, repair, loss, wage}` | 弹出结算面板 |
| `hide_settlement` | — | 关闭结算面板 |

## 8. 目录结构规范

```
godot 项目根/
├── global/                 # Autoload 单例（每个单例一个脚本）
│   ├── event_bus.gd
│   ├── sound_manager.gd
│   └── game_state.gd
├── scripts/                # 纯逻辑脚本（按领域分子目录）
│   ├── base_state.gd          # 框架基类放 scripts/ 根
│   ├── base_state_machine.gd
│   ├── devices/               # 设备：base_device.gd, generator.gd ...
│   ├── day/                   # 一天的流程状态：day_working.gd ...
│   ├── entities/              # 玩家、猴子等可动实体：player.gd ...
│   ├── world/                 # 场景物件：booth.gd ...
│   └── ui/                    # 界面逻辑：hud.gd ...
├── scenes/                 # 场景文件 .tscn
│   └── main.tscn
├── params/                 # 数据资源 / 配置（正式版的设备参数表等）
└── assets/                 # 美术与音频素材
    ├── animation/
    ├── font/
    ├── image/
    ├── shaders/
    └── sound/
```

放置规则：
- **脚本和它的场景就近**：可复用实体（玩家、猴子、某设备）做成独立 `.tscn`，脚本放 `scripts/对应子目录`，场景放 `scenes/`（或与实体同名子目录）。
- **框架基类**（`base_*.gd`）放 `scripts/` 根目录。
- **新领域**新建子目录，不要把所有脚本堆在 `scripts/` 根。
- **数据/配置**（正式版的数值表、关卡配置）放 `params/`。
- 一个文件只放一个 `class_name`。

## 9. 场景 / 节点规范

- 场景文件名 `snake_case`，根节点名 `PascalCase` 且语义化（如 `Player`、`Generator`）。
- 节点树层次清晰，按职责分组：`World`（世界物件）、`UI`（CanvasLayer 界面）、`DayManager`（逻辑）等。
- **可复用的东西做成场景**（PackedScene），用 `preload`/`@export PackedScene` 实例化，不在代码里手搭复杂节点树。
- UI 一律放 `CanvasLayer` 下（不随相机移动）。
- 脚本挂在它**负责**的节点上，不写“上帝脚本”管一切。

## 10. 资源命名

- 图片：`snake_case`，带用途前缀，如 `player_idle.png`、`icon_repair.png`。
- 音频：`snake_case`，按类别，如 `sfx_monkey_step.wav`、`sfx_generator_overheat.wav`。
- `SoundManager.SOUNDS` 的 key 用 `snake_case`，与文件名呼应：`"monkey_step"`。

## 11. Git 约定

- 分支：用 `script/new-branch.ps1` 创建，命名形如 `<用户名缩写>-dev`（如 `htt66g-dev`）。
- **不要在仓库子目录内运行切分支脚本**（Windows 会锁目录导致切换失败），在仓库根运行。
- 提交信息用中文、简短说明做了什么（如 `新建分支脚本`）。
- 每个功能/模块尽量独立分支，做完合并。

## 12. 提交前代码检查（强制）

本仓库配置了 **pre-commit 钩子**：每次 `git commit` 会对**本次暂存的 `.gd` 文件**自动跑格式检查 + 静态检查，**不通过就拒绝提交**。

### 一次性启用（每个克隆都要做一次）
```powershell
# 1. 安装检查工具（gdtoolkit 含 gdformat + gdlint）
pip install "gdtoolkit==4.*"

# 2. 启用钩子（钩子脚本在 script/hooks/，纳入版本控制、全员共享）
.\script\setup-hooks.ps1
# 等价于： git config core.hooksPath script/hooks
```

### 检查内容
| 步骤 | 工具 | 检查 |
|---|---|---|
| 1 | `gdformat --check --line-length 100` | 格式是否符合规范（缩进/空格/换行…） |
| 2 | `gdlint` | 命名、定义顺序、未用参数等（规则见 `.gdlintrc`） |

### 没通过怎么办
- **格式问题**：运行 `gdformat --line-length 100 <文件>` 自动修复 → `git add` → 重新提交。
- **静态检查问题**：按 gdlint 提示手动修（如函数名不合规、`const` 要在 `@export` 前、参数没用要加 `_` 前缀）。
- 确需强制跳过：`git commit --no-verify`（**不推荐**，会绕过审查）。

### 相关文件
- `script/hooks/pre-commit` —— 钩子脚本（bash，Windows 下经 Git Bash 运行）
- `script/setup-hooks.ps1` —— 一键启用脚本
- `.gdlintrc` —— gdlint 规则配置（行宽 100、Tab、成员顺序、命名规则等）
